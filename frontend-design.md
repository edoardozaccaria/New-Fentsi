# Frontend Design Skill

## Core Philosophy

Before writing a single line of code, establish a clear visual direction. Every UI should feel like it was designed by a human with taste, not generated by an algorithm.

## Typography Rules

**BANNED FONTS** — never use these defaults:

- Inter
- Roboto
- Arial
- Helvetica Neue (as default body)
- System-ui (without intentional pairing)

**USE INSTEAD** — pick one direction and commit:

- **Editorial / Bold**: Playfair Display, Libre Baskerville, DM Serif Display
- **Modern Geometric**: DM Sans, Space Grotesk, Outfit
- **Expressive**: Fraunces, Cabinet Grotesk, Clash Display
- **Tech / Mono accent**: JetBrains Mono, Fira Code (as accent, not body)

Font sizing should be intentional: large headings (48px+), clear hierarchy, generous line-height (1.5–1.75 for body).

## Color Palette Rules

Do NOT use generic Tailwind blue-500 defaults. Every project needs a real palette:

1. Define a primary brand color first (warm, cool, or neutral bias)
2. Build a complementary accent (not just a darker shade)
3. Use a thoughtful neutral (warm grays, cool slates — pick one direction)
4. Define semantic colors: success, warning, error, info

**Example palettes to draw inspiration from:**

- Deep jewel tones + warm cream (editorial luxury)
- Electric neon on near-black (tech/startup)
- Muted pastels + strong contrast type (soft modern)
- Earth tones + bold accent (trust + approachability)

## Design Direction — Mandatory Pre-Code Step

Before writing any component, answer these questions:

```
1. What is the visual mood? (editorial, playful, minimal, bold, technical, warm)
2. Who is the target user? (their aesthetic expectations)
3. What is the primary action? (design should funnel toward it)
4. What is one visual risk we'll take? (something that makes it memorable)
```

## Layout Principles

- **No generic card grids** — use asymmetry, overlapping elements, or strong visual rhythm
- **Whitespace is not empty space** — it's a design element, use it aggressively
- **One dominant visual per screen** — hero image, large stat, bold heading
- **Micro-interactions matter** — hover states, transitions (200-300ms ease), focus rings

## Component Standards

### Buttons

- Primary: strong contrast, clear label, generous padding (12px 24px min)
- Never use generic gray for disabled — use opacity-50 on the primary color
- Add subtle hover transforms (translateY(-1px) or scale(1.02))

### Forms

- Labels above inputs, never placeholder-only
- Clear focus states (ring, not just border-color)
- Inline validation with meaningful messages

### Cards

- Pick ONE card style per project and stick to it
- Either use shadow OR border OR background-color to separate — not all three
- Cards with images: always define aspect-ratio

## Responsive Design

- Mobile-first: design for 375px width first
- Breakpoints: sm (640), md (768), lg (1024), xl (1280)
- Navigation: hamburger at md and below
- Typography scales: reduce heading sizes at mobile by ~25%

## Dark Mode

If implementing dark mode:

- Don't just invert colors — rethink contrast ratios
- Backgrounds: use dark-900 (not pure black #000)
- Cards in dark mode: dark-800 (subtle elevation via lightness, not shadow)

## Anti-Patterns to Avoid

- Blue hero sections with white text (overdone)
- Stock photo backgrounds with overlay text
- Centered everything (use intentional alignment)
- Inconsistent border-radius (pick 4px, 8px, or 16px and stick with it)
- Gradients on gradients
- Too many font weights (max 3 per project: regular, medium, bold)

## Quick Start Checklist

Before shipping any UI:

- [ ] Typography is intentional (not a default)
- [ ] Color palette has been defined (not ad-hoc)
- [ ] Mobile version reviewed
- [ ] Primary CTA is visually dominant
- [ ] Loading/empty/error states exist
- [ ] Hover states on all interactive elements
